Anima uses a hybrid retrieval pipeline that combines three search strategies to surface the most relevant memories.

## Vector Search

Queries are embedded and compared against stored memory embeddings via cosine similarity. This captures semantic meaning — a query about "weekend activities" will match memories about "hiking on Saturdays" even without shared keywords.

Anima supports multiple embedding backends:

- **Local** (default) — BGE-M3 via ONNX (1024d). Runs offline, no API calls, no cost.
- **OpenAI-compatible** — OpenAI, Cohere, Voyage, Together, vLLM, or any provider with an `/embeddings` endpoint.

See the [quickstart](/quickstart#embedding-backend) for setup.

## Multi-Vector Search

Anima supports multiple named vector embeddings per memory, each derived from different source text (e.g., raw content vs. a summary). Configure named vectors via `[[vectors]]` in `config.toml`:

```toml
[[vectors]]
name = "content"
source = "content"
weight = 0.7
populate = "ingestion"

[[vectors]]
name = "summary"
source = "summary"
weight = 0.3
populate = "async"
```

When multiple vectors are configured, search queries all vec tables and blends per-memory results using a weighted average. Query routing heuristics automatically adjust weights: short factual queries boost summary vectors, long narrative queries boost content vectors. All vectors always participate — routing shifts emphasis, never excludes.

If no `[[vectors]]` section is present, a single implicit `content` vector is used and search behaves identically to before.

Vector status is available at `GET /api/v1/vectors/status`.

## Keyword Search

Full-text search provides exact-match recall. This ensures that specific names, identifiers, or technical terms are not missed by the vector search.

## Temporal Ranking

Recency matters. Temporal weighting lets you bias results toward more recent memories or toward older ones, depending on your use case. The `temporal_weight` parameter in search requests controls this signal.

## Search Modes

The `search_mode` parameter on the search endpoint controls which strategies are combined:

- **hybrid** (default) — combines vector, keyword, and temporal signals
- **vector** — vector similarity only
- **keyword** — keyword matching only
- **ask_retrieval** — runs the full `/ask` multi-stage pipeline (keyword expansion, entity resolution, temporal supplement, episode expansion, entity-linked retrieval) but returns ranked search results instead of an LLM-generated answer

## Result Grouping

Set `group_by` on a search request to deduplicate results by a memory field. Valid values: `episode_id`, `category`, `source`.

```json
{"query": "weekend plans", "limit": 10, "group_by": "episode_id"}
```

When grouping is active, only the highest-scoring result per group is returned. Each result includes `group_size` (how many memories were in that group) and `group_value` (the field value). Memories with null values in the grouped field form their own group.

Grouping applies only to the search endpoint — `/ask` and `/chat` retrieve all relevant memories for context assembly.

## Query Rewriting

Set `query_rewrite: true` on any search request to enable keyword expansion. This extracts content words from natural-language queries and runs additional searches, merging results with the original. It's the same zero-LLM keyword extraction used internally by `/ask`, now available on the basic search endpoint.

## Enriched Retrieval

Both `/api/v1/ask` and `/api/v1/chat` use an enriched retrieval pipeline that goes beyond a single search:

1. **Keyword expansion** — extracts content words from the query and runs additional searches
2. **Entity resolution** — detects names, resolves aliases, and searches by canonical identities
3. **Temporal supplement** — detects date references in the query and pulls matching memories
4. **Episode expansion** — fetches co-episode memories for multi-hop context
5. **Entity-linked retrieval** — traverses the knowledge graph to find memories connected to mentioned entities
6. **Score-based filtering** — drops results below 25% of the top score to reduce noise

You can access this pipeline directly via `search_mode: "ask_retrieval"` on the search endpoint.

## Cross-Encoder Re-Ranking

After initial retrieval, the top-10 candidates are re-scored by a cross-encoder model (bge-reranker-v2-m3 INT8) to improve precision. The re-ranker is enabled by default and can be configured via the `[reranker]` section in `config.toml`. When the initial retrieval already produces high-confidence results, re-ranking is skipped to avoid unnecessary latency.

## Auto-Tuned Hybrid Weights

The balance between vector and keyword search is auto-tuned from calibration observations. Every 5 minutes, the system analyzes retrieval outcomes and adjusts `weight_vector` and `weight_keyword` if the data shows a significant shift (>5%). Current weights are visible at `GET /api/v1/calibration/weights`.

## Confidence-Aware Scoring

Each memory carries a `confidence` value derived from its `source`. Confidence is factored into the final ranking score so that user-stated facts are weighted higher than inferred or observed memories, all else being equal.

## Conflict Detection

When `/ask` retrieves memories that have supersession history (one memory replaced another via correction or reconsolidation), the response includes a `conflicts` array showing what changed. This lets agents surface "Note: you previously said X but later said Y" when relevant.
